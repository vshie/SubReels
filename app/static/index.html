mdi-information-outline What SubReels is for

SubReels is a BlueOS extension that records video onboard the vehicle, on demand or on a schedule, and burns live MAVLink-derived telemetry into a .ass subtitle sidecar alongside each recording. It is aimed at the niche where the camera feed cannot be sent to the surface in real time — drop-cameras, AUVs, and other tetherless platforms where the recording must be saved on the vehicle itself.

Two streams are captured in parallel when both sources are available:

  • USB H.264 from an exploreHD camera on /dev/video2
  • RTSP H.265 from a RadCam at rtsp://admin:blue@192.168.2.10:554/stream_0 (UDP transport; h265parse config-interval=-1 re-inserts VPS/SPS/PPS on each IDR)

If /dev/video2 is not present at start time, the USB recording is silently skipped and the RTSP stream still records. Recording state and per-stream liveness are reported on the Recording tab and via the /status endpoint.

Adjacent extensions. Blue Robotics' DashCam (Willian Galvani) auto-records any RTSP stream when the vehicle is armed and is available in the BlueOS Extensions Manager without manual install. BlueOS 1.5+ ships a similar “record on arm” feature with Foxglove playback. SubReels exists for the case where you need explicit start/stop control from a script (or an external service) instead of tying recording to ARM state.
mdi-camera-iris Camera setup gotchas
Do not add /dev/video2 to the BlueOS Video Streams page. SubReels needs exclusive access to the USB camera while recording, so configuring it as a Video Stream in BlueOS will prevent the extension from opening it (and vice versa). As a result, you will not see a Cockpit preview from the USB camera while this extension is installed. This is by design — for a tetherless dive there is no surface to preview to anyway.

The RTSP H.265 endpoint is currently hardcoded to rtsp://admin:blue@192.168.2.10:554/stream_0 in app/main.py (RTSP_H265_ENDPOINT at the top of the file). If your IP camera lives at a different address, uses different credentials, or speaks a different RTSP path, edit that constant and rebuild the image.

mdi-script-text-play-outline Enabling Lua scripting on the autopilot

ArduPilot's Lua engine is disabled by default. To run a control script on the autopilot (e.g. the VP-AUV dive state machine in HAUV.lua), flip a single parameter:

  1. Open BlueOS → Vehicle Setup → Autopilot Parameters.
  2. Set SCR_ENABLE = 1.
  3. Click the yellow save and reboot button.

Drop your .lua files into the ardupilot_logs → firmware → scripts directory in BlueOS via the file explorer in Pirate Mode. ArduPilot loads them on autopilot boot; restart the autopilot after editing to pick up changes. Lua errors surface as GCS messages in Cockpit or QGroundControl notification area with the offending line number, which makes the edit → restart autopilot → observe in SITL iteration loop fast.

Test the dive logic in SITL first. The repo's HAUV.lua exposes HOVER_SIM_MODE = 1 which simulates the physical start switch so the full state machine (countdown → descend → bottom-strike → ascend-to-hover → surface) runs end-to-end on a desk before you trust the system at depth.
mdi-api HTTP control endpoints port {{ servicePort }}

SubReels listens on TCP port 5423 on the BlueOS host (host networking, so it is reachable as localhost:5423 from the autopilot, the Pi, or any container on the same vehicle network). Every endpoint is a simple HTTP GET that returns JSON.

mdi-language-lua Driving SubReels from a Lua script

ArduPilot Lua does not ship a high-level HTTP client, but it does expose a raw TCP socket via Socket(). Reaching SubReels is therefore a matter of opening a socket to the BlueOS host on port 5423 and sending a one-line HTTP/1.1 request. The block below is lifted directly from HAUV.lua in this repository: 

-- Trigger SubReels: HTTP GET /start over a raw TCP socket.
-- Mirrors the start_video_recording / stop_video_recording functions in HAUV.lua.
HTTP_HOST = "localhost"
HTTP_PORT = 5423

function start_video_recording()
    local sock = Socket(0)            -- 0 = TCP
    if not sock:bind("0.0.0.0", 9988) then
        gcs:send_text(6, "Failed to bind socket")
        sock:close(); return false
    end
    if not sock:connect(HTTP_HOST, HTTP_PORT) then
        gcs:send_text(6, string.format("Failed to connect to %s:%d", HTTP_HOST, HTTP_PORT))
        sock:close(); return false
    end
    local req = "GET /start HTTP/1.1\r\nHost: localhost\r\nConnection: close\r\n\r\n"
    sock:send(req, string.len(req))
    sock:close()
    return true
end

-- ...later, inside the dive state machine:
if depth > recording_depth:get() and is_recording == 0 then
    if start_video_recording() then is_recording = 1 end
end

stop_video_recording() mirrors the above with GET /stop. The full annotated state machine — countdown, descent with dynamic throttle adjustment, bottom-strike detection, altitude-hold during the hover, and surfacing — is in HAUV.lua at the repo root.

mdi-tune-variant VP-AUV mission parameters

If you are replicating the Vertical Profiling AUV drop-cam mission profile, the companion Lua script (HAUV.lua) registers a HOVER_* parameter family under ArduPilot. These parameters are not part of SubReels itself — they live on the autopilot and configure the dive — but they are listed here for convenience because every dive parameter from countdown to bottom-strike to hover time can be tuned through BlueOS → Vehicle Setup → Autopilot Parameters without editing code. Going from a 60 m test dive to a 410 m deep dive is a single change to HOVER_T_DEPTH.
